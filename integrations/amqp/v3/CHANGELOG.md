## [v3.0.0] - 2026-06-09 - Initial Release — RabbitMQ Message-Bus Transport, Auto-Reconnect, and Server-Sent Events Backplane

### Added

- Initial Melody v3 binding of the AMQP integration — a durable `messagebus/contract.Transport` backed by RabbitMQ (AMQP 0-9-1) on `rabbitmq/amqp091-go`. Developed v3-first; v1 and v2 bindings to follow.
- `service_resolver.go` — plug-and-play container registration: `RegisterConnectionService(registrar, connection)` (registers the broker connection under `ServiceConnection`, with `ConnectionMustFromResolver`/`ConnectionMustFromContainer`) and `RegisterTransportService(registrar, name, transport)` (with `TransportMustFromResolver`), so an application configures the connection once and resolves it — or a named transport — from many services. The README now documents the full publisher and consumer wiring end-to-end, how to run the consumer, and how routing keys work.
- `connection.go` — `Provider` with `NewProvider(...ProviderOption)` builder (`WithHeartbeat`), `Open(dsn) (*amqp091.Connection, error)`, and `Close`. DSN credentials are redacted in error context.
- `registry.go` — `MessageRegistry` with generic `RegisterMessage[T](registry, name)`; maps message Go type ↔ wire type name so the consumer can reconstruct the concrete type from the `x-message-type` header; a conflicting re-registration (the same name bound to a different type, or vice versa) panics instead of silently overwriting and corrupting later deserialization.
- `transport.go` — `Transport` / `TransportConfig` implementing `Send`, `Receive`, `Ack`, `Nack`, `Close`. Lazily opens separate publish and consume channels, declares queue/exchange topology on first use, JSON-serializes the message body (reusing core `serializer.JsonSerializer`), and sets prefetch via QoS. Decode failures are dead-lettered. On `Nack(requeue=true)` the message is re-published (not broker-requeued) carrying an incremented `x-redelivery-count` header, which `decode` reads back into a `messagebus.RedeliveryStamp` — so the consumer's `RetryPolicy` (max retries, dead-letter transport) is honored uniformly across transports instead of relying on the broker's one-shot `redelivered` flag. A `DelayStamp` re-publishes through a `<queue>.delay` queue (per-message TTL, dead-lettered back to the main queue), so the consumer's backoff actually spaces out retries. Consume-channel acknowledgements (`Ack`/`Nack`/the requeue re-publish, and the decode-failure nack emitted from the delivery-forwarding goroutine) are serialized through a dedicated consume mutex, mirroring the publish mutex, since an `amqp091` channel is not safe for concurrent use. The `x-redelivery-count` header is read tolerantly across broker encodings (`int*`/`uint*`/`float*`) so a count round-tripped as a float never silently resets the poison-message guard; the publish path refuses to re-open a channel on a closing transport; and the re-publish-then-ack requeue is documented as at-least-once (a crash between the publish and the ack can redeliver the original alongside the re-published copy, so handlers must be idempotent).
- `transport.go` — optional auto-reconnect. Setting `TransportConfig.Dialer` (e.g. `Provider.Dialer(dsn)`) lets the transport re-establish the broker connection after it drops: the consumer's delivery loop, on losing its channel, re-dials and re-subscribes with bounded exponential backoff (1s→30s) and resumes on the same output channel, so the message-bus consumer survives a broker restart without an external supervisor. A single redial is in flight at a time, the publish path drops a dead channel and retries once, and `Close` tears the reconnect loop down and closes only connections the transport itself dialed (never the caller-provided one). When `Dialer` is nil the previous behaviour is preserved exactly (a dropped connection stops the consumer). Each delivery is stamped with the consume-channel generation it arrived on, and `Ack`/`Nack` skip — as a no-op — any acknowledgement whose generation no longer matches the live channel, so a message received just before a reconnect can never have its stale delivery tag acked against the rotated channel (which would otherwise ack an unrelated delivery and silently lose it at `prefetch>1`, or trip a 406 channel close → reconnect churn at `prefetch=1`); the broker redelivers the still-unacked message, so handlers must be idempotent (already required by the at-least-once requeue). Behind a load balancer the consumer runs as competing consumers across instances (normal AMQP fan-out), so redelivery may resume on a different instance — idempotency must be keyed on the message.
- `connection.go` — `Provider.Dialer(dsn)` returns a `func() (*amqp091.Connection, error)` that re-opens the connection from the same dsn, for `TransportConfig.Dialer`.
- `server_sent_event_backplane.go` — `NewServerSentEventBackplane(ServerSentEventBackplaneConfig{Connection|Dialer, Hub, Exchange, Logger})` is a fanout backplane for the core `http.ServerSentEventHub`, so Server-Sent Events/WebSocket broadcasts fan out across every application instance behind a load balancer. Each broadcast is published to a fanout exchange (default `melody.sse`) tagged with a per-instance random origin; each instance binds its own exclusive, auto-deleted queue, forwards other instances' events into the hub via `DeliverLocal`, and ignores the echo of its own origin. Replication is best-effort (auto-ack, transient). With a `Dialer` the subscription and publisher re-establish after a broker restart with bounded backoff; `Close` tears the subscription down and closes only a connection the backplane itself dialed.
- `stamp.go` — `DeliveryStamp` (`Tag`, `Redelivered`, `Generation`) carried on received envelopes so `Ack`/`Nack` can map back to the AMQP delivery; `Generation` records the consume-channel epoch so a post-reconnect acknowledgement of a stale tag is detected and skipped.
- Dead-letter support: when `DeadLetter` is enabled, declares `<queue>.dlx` / `<queue>.dlq` and points the main queue at it. A message whose retries are exhausted is nacked without requeue (`Nack(requeue=false)`) so it lands in the DLX rather than being dropped; the retry count itself is owned by the consumer through the persisted `x-redelivery-count` header (see `transport.go`).
- `parameter.go` — `ParameterDsn`, `ParameterExchange`, `ParameterPrefetch`, and `RegisterDefaultParameters`.
- `transport_test.go` — send/receive/ack integration test, plus tests that a re-published message persists its `x-redelivery-count` and dead-letters once retries are exhausted, that a `DelayStamp` routes through the `<queue>.delay` queue and returns after its TTL, and that the consumer reconnects after its broker connection is dropped and still delivers a subsequently-published message on the same channel; all skipped unless `AMQP_DSN` is set; verified end-to-end against RabbitMQ 3.13.
- `reconnect_test.go` — broker-free unit tests for the reconnect machinery: backoff growth/cap, the no-dialer fallback, single-flight redial, the delivery-loop lost/done reasons, and that the consume loop closes its output channel on both channel-loss-without-dialer and context cancellation.
- A `rabbitmq` service added to `.dev/docker/docker-compose.yml` for local/integration runs.

### Fixed

- The poison-message (decode-failure) log now states whether the message is being dead-lettered or dropped based on whether `DeadLetter` is configured — when no DLQ is declared a nacked-without-requeue delivery is discarded by the broker, so the log no longer misleadingly claims "dead-lettering". Enable `DeadLetter` in production so undecodable deliveries are retained.
- `server_sent_event_backplane.go` — the reconnect paths now close the previous consume/publish channel before overwriting it (`subscribe` re-subscribing onto a fresh channel, and `liveConnection` redialing the connection), so a channel-only loss that leaves the connection alive no longer leaks the stale `*amqp091.Channel`.
- `server_sent_event_backplane.go` — `Publish` now retries once after resetting a dead publish channel, mirroring the message-bus transport's publish path: a broadcast that lands on a silently-closed channel (the broker dropped the channel but the connection is alive) is re-published on a freshly opened channel instead of being lost until the next broadcast happens to re-open it. The retry is single and bounded (best-effort Server-Sent Events, no amplification) and is skipped when the backplane is closing.
- `server_sent_event_backplane.go` — `Close` now detaches the backplane from the hub (`SetBackplane(nil)`) before tearing down, so a `Broadcast` issued after `Close` is no longer replicated to a torn-down backplane on a cancelled context (which would otherwise repeatedly fail the publish and reset the dead channel on every subsequent broadcast).
- `transport.go` — `Transport.Close` now unblocks a consume goroutine parked on the output channel. `forwardDeliveries` only released on the runtime context being cancelled, so calling `Close` directly (without cancelling the context) while a decoded delivery was waiting for a reader on the output channel leaked the consume goroutine. `Close` now closes an internal signal that both `forwardDeliveries` selects observe, so the goroutine returns promptly on a direct `Close`.
- `transport.go` — `ensurePublishChannel`/`ensureConsumeChannel` now treat a cached-but-closed `*amqp091.Channel` as absent (`IsClosed()` guard on both the fast path and the post-open double-check) and open a fresh one. A broker channel-level exception (publishing to a removed exchange, a policy violation, any protocol error) closes the channel while leaving the connection alive; with no `Dialer` configured (a supported `Connection`-only transport) the publish retry never reset the channel, so the closed channel was reused forever and every subsequent `Send` failed permanently even though a fresh channel could be opened on the still-live connection. The guard restores self-healing for channel-level losses without requiring a reconnect.
