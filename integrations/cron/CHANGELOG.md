## [v1.0.0] - 2026-05-16 - Initial Release — Cron Integration

### Added

- Initial Melody v1, v2, and v3 bindings of the cron integration. Each binding is its own Go module and depends on the matching melody version; the three packages share the same exported API and behavior, differing only in the melody import paths. v1 lives at the module root (`github.com/precision-soft/melody/integrations/cron`, following Go's no-suffix convention for v0/v1), v2 at `./v2/`, v3 at `./v3/`
- `schedule.go` — pure 5-field `Schedule` struct (`Minute`, `Hour`, `DayOfMonth`, `Month`, `DayOfWeek`); `Schedule.Defaults()` (in-place wildcard fill, nil-safe) and `Schedule.Expression()` (renders 5-field crontab string, auto-fills empty fields with `*` without mutating the receiver)
- `configuration.go` — declarative cron configuration decoupled from CLI commands:
    - `EntryConfig` — per-command settings: `Schedule *Schedule` (time fields), `User`, `LogFileName`, `LogFileNameRaw bool`, `LogDisabled bool`, `DestinationFile`, `Command []string` (full argv override), `Instances int` (1-N multi-entry expansion; values `< 1` normalize to `1`)
    - `ScheduledCommand` — pairs `CommandName` with `*EntryConfig`
    - `Configuration` — ordered registry built via `NewConfiguration().Schedule(commandName, config).Schedule(...)`; preserves insertion order so generated crontab files are stable
    - `CommandName[T clicontract.Command](factory func() T) string` — generic helper that instantiates a command constructor and returns its `Name()`, letting userland reference commands by constructor rather than hardcoded string
- `generate_command.go` — `GenerateCommand` constructed via `NewGenerateCommand(configuration)`, exposes `melody:cron:generate`; flags `--out`, `--logs-dir`, `--binary`, `--user`, `--heartbeat-path`, `--heartbeat-command` (repeatable, one argv token per occurrence — overrides `--heartbeat-path`), `--heartbeat-destination` (repeatable, restricts the heartbeat to specific destinations — accepts `default`, an absolute path, or a relative path matched against `dir(--out)`; misspelled values error out); preserves registration order of entries within each destination file; groups entries by `EntryConfig.DestinationFile` and writes one file per distinct destination (relative `DestinationFile` values that escape `dir(--out)` are rejected); appends the configured heartbeat to every generated file by default; honors `EntryConfig.LogDisabled`; emits `nothing to write` and skips file creation when the configuration is empty and no heartbeat is provided; emits
  `wrote heartbeat-only crontab to <path>` for heartbeat-only outputs. Each flag cascades from CLI override → container parameter (no hardcoded fallback inside the command — defaults live in the parameter system via `RegisterDefaultParameters` or user-supplied parameters). Errors out when output path, logs-dir (with logging-enabled commands), or user are missing after the cascade. Binary falls through to `os.Executable()` only when at least one scheduled command needs the default `<binary> <name>` shape — configs where every `Schedule` sets a custom `Command` skip the lookup so the generator works even when the running binary has been deleted. Multi-instance log file names preserve compound extensions (`archive.tar.gz` + `Instances = 2` → `archive-1.tar.gz`, `archive-2.tar.gz`)
- `template.go` — minimal abstraction file: `Template` interface (`Name() string`, `Render(entries, options) (string, error)`), `BuiltinTemplates() []Template` (returns the slice of in-tree templates so `NewGenerateCommand` auto-registers them without enumerating each one by hand), and a free `Render(entries, options)` wrapper retained as a thin shim over `(&CrontabTemplate{}).Render(...)` for back-compat. Each concrete template lives in its own file so adding new formats is a matter of adding `template_<format>.go` next to `template_crontab.go`
- `template_crontab.go` — in-tree crontab implementation: `CrontabTemplate{}` (registered under the registry-key constant `TemplateNameCrontab = "crontab"`), `Render(entries, options)`, and the private `buildCrontabLine` line builder. `Render` rejects entries with empty `User`, with both empty `Binary` and no `EntryConfig.Command` override, or with a heartbeat (path or command) missing a `HeartbeatUser`. `Entry` carries the user-provided `Schedule` (which exposes the `Command` and `DestinationFile` per-command overrides); `RenderOptions` exposes `HeartbeatPath` for the simple `/bin/touch` heartbeat and `HeartbeatCommand` for fully custom heartbeat command parts (takes precedence over `HeartbeatPath`). Applies POSIX shell quoting **per token** to `EntryConfig.Command`, `Entry.Binary`, `Entry.Args`, `RenderOptions.HeartbeatCommand`, `LogPath`, and `HeartbeatPath` — tokens with spaces, single quotes, or other shell metacharacters are wrapped in single quotes (embedded `'` escaped as
  `'\''`); safe tokens render unchanged. Generic `ForbiddenChar` type and `ValidateNoForbiddenChars(tokens, forbidden, context)` helper expose the per-template forbidden-char validation; `CrontabForbiddenChars` ships the crontab-specific list (currently `%`, which crond treats as a line-continuation character before the shell sees it — single-quoting alone does not protect it). Custom templates supply their own slice of `[]ForbiddenChar` matching the constraints of their output format
- `generate_command.go` — `GenerateCommand.RegisterTemplate(template)` adds a `Template` to the command's registry (keyed by `Template.Name()`); passing a template whose `Name()` collides with a built-in replaces it for that command instance. `NewGenerateCommand` auto-registers everything `BuiltinTemplates()` returns. The active template is resolved from the `--template` flag → `melody.cron.template` parameter → fallback `"crontab"`. Unknown names error out with a message listing the registered names. Adds the `--template` flag to `Flags()`
- `parameter.go` — exported parameter-name constants `ParameterUser`, `ParameterLogsDir`, `ParameterBinary`, `ParameterDestinationFile`, `ParameterHeartbeatPath`, `ParameterTemplate` (namespace `melody.cron.*`) that user modules register with the application's `ParameterRegistrar` to supply defaults for `melody:cron:generate`; `ParameterRegistrar` interface (structural — `melodyapplicationcontract.ParameterRegistrar` satisfies it implicitly); `RegisterDefaultParameters(registrar)` helper that registers only the integration's safe, environment-agnostic defaults: `melody.cron.destination_file = %kernel.project_dir%/generated_conf/cron/crontab`, `melody.cron.logs_dir = %kernel.logs_dir%/cron`, `melody.cron.template = crontab`. `melody.cron.user` is intentionally **not** registered by the helper — there is no portable default OS user, and silently shipping `www-data` would mask misconfigured deployments. Userland supplies it via `--user`, `melody.cron.user`, or per-command
  `EntryConfig.User` (or generation fails with `ErrEntryEmptyUser` / `ErrHeartbeatUserMissing` and an actionable message). `kernel.project_dir` resolves to the source root under `go run` and to the binary's directory in a compiled build, so the same defaults work for both modes
- `README.md`, `v2/README.md`, `v3/README.md` — umbrella usage guide (configuration parameters, cascade rules, per-command overrides, custom heartbeat command, heartbeat per-destination targeting, template customization, footguns) plus per-version install/import snippets. New umbrella "Customizing the template" section walks through the `Template` interface, `RegisterTemplate`, built-in template auto-registration, template-name resolution priority, and template-specific configuration patterns
- `.example/` (v1, v2, v3 with version-matched import paths) — copy-paste reference module split into four single-purpose files: `command.go` (a plain Melody CLI command `BillingCleanupCommand`), `cron_configuration.go` (a `cron.Configuration` registry that maps the command name to an `EntryConfig`), `template_k8s.go` (a custom Kubernetes CronJob `Template` implementation with its own `ForbiddenChar` list), and `module.go` (the melody `Module` wire-up: `RegisterParameters` + `RegisterCliCommands` + `RegisterTemplate`, plus compile-time `applicationcontract.Module` / `ParameterModule` / `CliModule` assertions). The directory is dot-prefixed so `go build ./...` and `go test ./...` skip it; build it explicitly with `go build ./integrations/cron/.example/`
- Atomic destination writes: each destination is written via `os.CreateTemp` to a uniquely-named sibling file (e.g. `crontab.123456789.tmp`), `fsync`-ed, chmod-ed to the requested mode, then renamed over the final path so `crond` never reads a half-written file even if the generator is killed mid-write. After `rename`, the destination directory is also `fsync`-ed so the directory entry survives an immediate crash on POSIX filesystems. Concurrent generator invocations against the same `--out` use distinct temp file names (`os.CreateTemp` randomization) and the temporary file is removed on any error before rename
- Uniform `cron: …` error prefix on every error emitted by `Render`/`buildCrontabLine`/`generate_command.go` so cron errors remain identifiable when surfaced alongside errors from other components
- `Schedule.Expression()` is nil-safe (returns `* * * * *` when called on a `nil` receiver), mirroring `Schedule.Defaults()`
- Whitespace rejection on `Schedule.Minute/Hour/DayOfMonth/Month/DayOfWeek` (space, tab, newline, carriage return) — would otherwise produce a malformed crontab line. Valid range/step/list notation (`*/15`, `1-5`, `1,2,3`, `mon-fri`, etc.) continues to be accepted
- `resolveEntryLogPath` rejects `EntryConfig.LogFileName` values whose resolved path escapes `--logs-dir` (e.g. `"../escape.log"`), mirroring the `EntryConfig.DestinationFile` guard
- `CrontabForbiddenChars` rejects `%`, `\n`, `\r` in any rendered token (`EntryConfig.Command`, `Args`, `LogPath`, `HeartbeatPath`, `HeartbeatCommand`); embedded newlines would otherwise slip past `singleQuote` and split one entry into multiple invalid lines
- `buildCrontabLine` rejects a non-empty `EntryConfig.Command` whose tokens are all empty strings (e.g. `[]string{"", ""}`), which would have rendered as `'' ''` and run the empty command at the scheduled time
- `validateUserField` rejects any user value (`Entry.User` resolved through the `EntryConfig.User` / `--user` / parameter cascade and `RenderOptions.HeartbeatUser`) that contains whitespace (space, tab, CR, LF). Such a value would split a crontab line and is rejected with `cron: ... contains whitespace; user fields must be single tokens`. Defense-in-depth — usernames come from trusted application code in practice
- Errors emitted by `generate_command.go`, `template_crontab.go`, and `validation.go` are now constructed via `exception.NewError(message, context, causeErr)` instead of `fmt.Errorf`, so the framework's JSON logger receives structured context (entry name, paths, parameter names, requested vs. resolved values, template name, …) alongside the human-readable message. The `cron:` prefix on every `.Error()` is preserved
- `runWithConfiguration` is split into three private helpers — `resolveRunOptions`, `collectScheduledEntries`, and `writeDestinations` — each unit-testable on its own; the top-level method is now a ~15-line pipeline
- Pre-flight check: when a heartbeat (path or command) is enabled but no user is resolved through the `--user` / `melody.cron.user` cascade, `melody:cron:generate` fails fast with `cron: heartbeat is configured but no user is set; …` instead of bubbling up the lower-level "heartbeat user is empty" error from `Render`
- `configurationFromRuntime` now surfaces container lookup failures instead of silently degrading to "CLI flags only"; the configuration service's resolution error is wrapped and propagated from `Run`, so users learn why their parameter cascade was bypassed
- `--out=""`, `--logs-dir=""`, `--binary=""`, `--user=""`, `--heartbeat-path=""`, `--template=""` are now treated identically to "flag not provided" — `resolveDefault` cascades through to the container parameter when the flag was set to an empty string, instead of short-circuiting to the empty CLI value
- `syncDir` no longer silently drops the close error when fsync succeeds, and wraps every failure mode with the directory path; when both `Sync()` and `Close()` fail the underlying errors are joined via `errors.Join`
- Package-level sentinel errors exported in `errors.go` — `ErrNoOutputPath`, `ErrNoLogsDir`, `ErrTemplateNotFound`, `ErrHeartbeatUserMissing`, `ErrHeartbeatDestinationUnmatched`, `ErrHeartbeatDestinationDefaultMissing`, `ErrDestinationEscape`, `ErrEntryEmptyUser`, `ErrEntryEmptyCommand`, `ErrForbiddenCharacter`, `ErrFieldContainsWhitespace`. Every configuration / validation error built by `generate_command.go`, `template_crontab.go`, and `validation.go` now wraps the matching sentinel as its `causeErr`, so callers can branch with `errors.Is(err, cron.ErrXxx)` without parsing the message. I/O failures (mkdir, rename, fsync, …) keep their original OS error as cause — those remain `errors.Is`-able against `os.Err*` sentinels as before

### Changed

- `cli/contract/type.go` (root + v2/ + v3/) gains `type StringSliceFlag = urfavecli.StringSliceFlag` so the cron command (which uses repeatable string-slice flags for `--heartbeat-command` and `--heartbeat-destination`) consumes `clicontract.StringSliceFlag` like every other flag in the integration, without an extra `urfavecli` import in `generate_command.go`
- `README.md` — added a "Cron expression validation" section that documents which checks the generator runs at generation time and which it deliberately leaves to the cron daemon at install time, with `crontab -T` recommended as the post-generation gate. Clarified the `EntryConfig.DestinationFile`-absolute-path semantics, documented the hardcoded `0644`/`0755` file modes, and added a new "Package surface" section listing every exported identifier. `v2/README.md` and `v3/README.md` now reference that section uniformly so they no longer cross-reference each other asymmetrically.

### Added (tests)

- `template_test.go` — direct unit tests for `ValidateNoForbiddenChars` (forbidden char rejection, clean tokens, custom forbidden list, nil tokens), `BuiltinTemplates`, `shellQuoteIfNeeded` (empty, safe tokens, space, metachar), `singleQuote` (embedded single quote escaping), `joinShellTokens`, the new whitespace validation for `Schedule` fields, and a positive test that valid range/step/list cron expressions continue to render.
- `template_test.go` — newline / carriage-return rejection tests for entry arg, log path, heartbeat path, heartbeat command, and `EntryConfig.Command`; plus `TestRenderRejectsScheduleCommandWithOnlyEmptyTokens`.
- `generate_command_test.go` — `TestRunPreservesRegistrationOrderWithinSharedDestination` locks in registration-order preservation when multiple entries share a non-default `DestinationFile`. `TestRunAtomicWriteHandlesConcurrentRuns` exercises five concurrent `melody:cron:generate` invocations against the same `--out` and asserts that the destination file is well-formed and no orphan `.tmp` siblings remain. `TestRunAtomicWriteLeavesNoTempFileOnSuccess` now scans the output directory for any `*.tmp` leftovers rather than checking a single predictable path. `TestRunRejectsLogFileNameEscapingLogsDir` covers the new `EntryConfig.LogFileName` path-escape guard.
- `log_filename_test.go` — new dedicated test file gathering the previously scattered `sanitizeLogFileName` / `rawLogFileName` / `splitLogFileExtension` tests, plus `TestSplitLogFileExtensionHiddenWithExtension`, `TestSplitLogFileExtensionEmpty`, and `TestRawLogFileNameReplacesSlashesButKeepsColons` for previously uncovered edge cases.
- `generate_command_test.go` — `TestRunInstancesZeroIsNormalizedToOne` and `TestRunInstancesNegativeIsNormalizedToOne` lock in the documented "`EntryConfig.Instances < 1` is normalized to 1" contract. `TestRunHeartbeatDestinationAbsolutePath` covers the previously untested absolute-path form of `--heartbeat-destination`.
- `template_test.go` — `TestRenderRejectsEntryUserContainingWhitespace`, `TestRenderRejectsEntryUserContainingNewline`, `TestRenderRejectsHeartbeatUserContainingWhitespace`, `TestRenderRejectsHeartbeatUserContainingNewlineForCommand` exercise the new `validateUserField` guard on both entry users and the heartbeat user (path and command branches).
- `generate_command_test.go` — `TestRunErrorsWhenHeartbeatEnabledWithoutUser` covers the new pre-flight heartbeat-user check, and `TestRunCascadesParameterWhenOutFlagIsEmpty` locks in the "empty flag string falls through to parameter cascade" semantics.
- `log_filename_test.go` — `TestSplitLogFileExtensionAllDots` documents the all-dots input fallback (`splitLogFileExtension("...") == ("...", "")`).
- `errors_test.go` — new dedicated test file with 14 `errors.Is`-based spot-checks, one per exported sentinel, covering both the generate-command surface and the lower-level `Render` / `ValidateNoForbiddenChars` / `validateUserField` / `validateScheduleFields` entry points.
